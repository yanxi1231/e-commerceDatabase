================================================================================
1. OVERALL VERDICT
================================================================================

    PASS WITH MINOR ISSUES

The project has a well-structured, self-contained SQL dump file (schema.sql)
containing 14 tables (exceeding the 9-table minimum), DDL, DML seed data,
3 triggers, 3 user-defined functions, 2 stored procedures, and 1 scheduled
event. Foreign keys with appropriate ON DELETE / ON UPDATE actions are present
on all relationships. The schema is in 3NF and constraints are well-applied.

Minor issues:
  (1) The stored procedures (sp_checkout, sp_add_to_cart) are defined in the
      dump but the Java application duplicates this logic in CheckoutService /
      CartService instead of calling them, slightly weakening the "modularity
      via server-side objects" argument.
  (2) The event scheduler (evt_cancel_stale_orders) requires
      SET GLOBAL event_scheduler = ON which is not included in the dump.

D. ON DELETE / ON UPDATE actions
--------------------------------------------------------------------------------
Status   : PASS
Evidence : All 15 foreign keys have explicit referential actions. Examples:
             - user_addresses(user_id)  ON DELETE CASCADE  ON UPDATE CASCADE
             - products(category_id)    ON DELETE RESTRICT ON UPDATE CASCADE
             - orders(user_id)          ON DELETE RESTRICT ON UPDATE CASCADE
             - cart_items(variant_id)   ON DELETE SET NULL ON UPDATE CASCADE
             - categories(parent_id)    ON DELETE SET NULL ON UPDATE CASCADE
Why      : Actions are sensible — CASCADE for owned child records, RESTRICT to
           prevent deleting referenced parents (users with orders, categories
           with products), SET NULL where appropriate (optional variant on cart
           items, parent category).

--------------------------------------------------------------------------------
E. 3NF / Redundancy review
--------------------------------------------------------------------------------
Status   : PASS
Evidence : No repeating groups, no duplicated facts, no partial or transitive
           dependencies observed. Key design decisions:
             - order_items.product_name_snapshot and price_at_time are
               intentional denormalization (historical snapshot at time of
               order) — correct practice, not a 3NF violation.
             - product_images and product_variants are properly separated
               from products.
             - inventory is a separate table from products, correctly
               normalized with a composite unique key
               (product_id, variant_id, warehouse_id).
Why      : The schema is in 3NF. The only denormalization
           (product_name_snapshot, price_at_time) is deliberate and correct
           for order history.

--------------------------------------------------------------------------------
Status   : PASS
Evidence :

      Atomic order creation with inventory management, stock validation,
      payment creation, and cart clearing inside a transaction.
    - sp_add_to_cart     (schema.sql lines 417-465)
      Adds/updates item in cart with stock validation.

  User-Defined Functions (3):
    - fn_available_stock   (lines 249-267)
      Calculates available stock for a product/variant across warehouses.
    - fn_effective_price   (lines 272-292)
      Calculates price with variant override.
    - fn_avg_rating        (lines 296-306)
      Calculates average product rating.

  Triggers (3):
    - trg_after_user_insert          (line 206)
      Auto-creates cart when a new user is inserted.
    - trg_before_user_delete         (line 218)
      Prevents deleting users with active orders via SIGNAL SQLSTATE '45000'.
    - trg_inventory_update_timestamp (line 234)
      Updates last_updated on inventory changes.

  Events (1):
    - evt_cancel_stale_orders        (line 474)
      Cancels pending orders after 30 minutes, runs every 5 minutes.

Why      : All four types of server-side objects are present with meaningful
           logic. The procedures include transaction management, stock
           validation, and error handling.

--------------------------------------------------------------------------------
H. Error handling
--------------------------------------------------------------------------------
Status   : PASS
Evidence :
  Database-side:
    - Both stored procedures have:
        DECLARE EXIT HANDLER FOR SQLEXCEPTION
        BEGIN ROLLBACK; SET p_result_msg = '...'; END;
      (schema.sql lines 331-335 and 429-431)
    - trg_before_user_delete uses:
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Cannot delete user with active orders';
      (schema.sql lines 226-227)

  Application-side:
    - All controller methods wrap operations in try/catch blocks with
      RedirectAttributes for user-facing error messages.
      (e.g., CheckoutController.java lines 57-73)
    - Service methods throw IllegalStateException / IllegalArgumentException
      for validation failures (empty cart, insufficient stock, not found).
    - @Transactional ensures rollback on unhandled exceptions.
    - GlobalControllerAdvice.java catches exceptions in cart count / user
      name resolution to avoid breaking page rendering.

Why      : Both database-level and application-level error handling are
           present and meaningful.

--------------------------------------------------------------------------------
I. Modularity / Excessive SQL in front end
--------------------------------------------------------------------------------
Status   : PASS WITH MINOR ISSUES
Evidence :
  - The project follows a clean Controller -> Service -> Repository
    architecture. Controllers contain zero SQL — they only call service
    methods (CartController.java, CheckoutController.java, etc.).
  - Services contain business logic but no raw SQL strings — they use
    Spring Data repository methods and JPQL queries defined in repository
    interfaces.
  - The 2 custom @Query annotations use JPQL (not raw SQL) and are properly
    isolated in repository interfaces:
      InventoryRepository.java lines 10-16
      ProductRepository.java lines 15-19
  - Minor issue: The Java CheckoutService.processCheckout() and
    CartService.addToCart() duplicate the logic of sp_checkout and
    sp_add_to_cart stored procedures rather than calling them via JDBC.
    This means the stored procedures exist in the dump but are not invoked
    at runtime by the application.

Why      : The application is well-layered with no SQL in controllers or
           templates. The stored procedures exist and work, even though the
           app also has its own transactional service layer.

4. RECOMMENDATIONS
================================================================================

1. Add "SET GLOBAL event_scheduler = ON;" at the top of schema.sql so the
   scheduled event actually fires on a fresh MySQL install.

2. (Optional improvement) Consider having the Java CheckoutService call
   sp_checkout via JDBC CallableStatement instead of duplicating the logic.
   This would more directly demonstrate that the stored procedures are used
   by the application. Currently they exist in the database but the Spring
   app uses its own equivalent Java logic.
